### 2026-06-14 — FIFO 预测加 per-office 校准系数 + 修两处显示/模型缺陷

**背景**：FIFO 上线后用户反馈「预计时间有点奇怪」。排查出三个问题。
**决定/修复**：
1. **日期截断塌缩**（bug）：`today.plusMonths(x.toLong())` 把月数截成整数，9.2/9.9 月都变 9 月，导致乐观/正常/保守三张卡里两张显示同一个月。改 `plusFractionalMonths`（30.44 天/月按天推算）。
2. **区间不对称**：「正常」原用最新单月处理量，东京最新月恰偏慢 → 正常≈保守、乐观偏远。改 `computeWait` 用近 6 月平均 `μ_central` 做中心估算 + gap 外推。Case A 的近 6 月均恰好=最新月（都 1000），故 38 测试全不受影响。
3. **新提交者低估**（模型缺陷）：提交月晚于最新数据月时，gap 外推错误扣掉数据延迟期处理量。改 gap 起点取 `max(最新数据月, 提交月)`，新提交者不再误扣。
4. **校准系数**：模型按 `积压÷处理量` 硬算东京约 14 月，但实际 ~700 天（~23 月）——队列非纯 FIFO、资源跨资格共享、部分案卡住。`OfficeData` 加 `calibrationFactor`（默认 1.0，数据驱动可月度调），东京设 **1.65**，使新提交者总时长 ≈ 698 天。校准只乘**时间**，不动 rNow（排队人数是真实计数）。
**理由**：用户在「数据硬算 vs 对齐 700 天」中选了校准对齐实际观测。系数是经验值、需说明，文案统一标「参考估算」。
**取舍**：校准只乘剩余时间 → 老提交者总时长 < 700 天（已走过一截，合理）；新提交者命中 700 天这个用户最关心的头条数字。其他局无 ground truth，暂留 1.0，待测。
**影响**：`FifoPrediction.kt` / `PredictionEngine.kt` / `PublicDataDoc.kt`（加字段）/ 数据脚本 `CALIBRATION` dict + 重新生成 `public-data.json`。

### 2026-06-14 — 核心逻辑引入单元测试，UI 仍跳过

**决定**：从预测/数据管线这块起，核心纯逻辑（`PredictionEngine` FIFO 算法、CSV→JSON 转换+地区拆分、降级链）写单元测试；UI（Compose）继续跳过。测试策略详见 `docs/PREDICTION_AND_DATA.md` §10。
**理由**：之前「测试暂跳过」针对的是 UI（Compose 测试成本高收益低）。但 FIFO 算法/数据拆分全是纯逻辑、边界多、算错肉眼看不出，出错代价高、又最好测，性价比与 UI 相反。`testImplementation(junit)` 已在，零配置可写。
**影响**：修正下方「瘦身流程」里的「测试步骤暂跳过」——改为「UI 跳过、核心逻辑必测」。实现 `PredictionEngine` 时算法与单测同步写。

### 2026-06-13 — 预测算法升级为 FIFO 排队模型 + e-Stat 真实数据（A2 托管）

**决定**：把处理时间预测从「固定 4-6 个月 + 拍脑袋 buffer」改为「基于入管局真实积压/处理量的 FIFO 排队消化模型」；数据走 **A2 方案**——GitHub raw 托管人工维护的 JSON、每月跟 e-Stat 月報更新、**不上后端**。完整方案见 `docs/PREDICTION_AND_DATA.md`。
**理由**：
- e-Stat「受理及び処理人員」表公开了 未済(积压)/新受/既済(处理) 按地区按月数据，足以建排队模型，比固定值精确得多且全可解释。竞品 xiaofushi.com 已用同一数据源（sid=0003449073）验证可行。
- 数据年/月度更新、量小，上后端定时抓取性价比极低（详见下方被否的 B 方案）；A2 用静态 JSON 托管让「人工更新」成本从「每月发版过审」降到「每月改个文件」，无后端逻辑。
**关键约束/取舍**：
- 「越来越准、永远最新」依赖数据持续更新；A2 下靠每月人工更新，漏更新预测会失真——相对竞品后端的固有差距，已接受。
- **东京 e-Stat 数据含横滨支局**，横滨 2025 起处理超快会把东京拉低，必须人工拆分（拆后东京实际约 700 天）。
- 数据约 3 个月延迟（正常）；国内访问 raw.githubusercontent / jsDelivr 不稳，**APK 内置兜底为必须**。
- 仍给不了个案实时进度（入管不公开），模型给的是「按平均你大概排到哪」，文案需说明为参考估算。
**影响**：新增 `docs/PREDICTION_AND_DATA.md`；后续重写 `PredictionEngine`、扩展 `staticdata/PublicData`、加网络层+三层降级、共享时间线、公开数据卡/数据页接真实数据。原「时间线+公开数据卡接真实数据」任务并入此方案。

### 2026-06-13 — 否决「后端定时抓取 e-Stat」(B 方案)

**决定**：不为公开数据单独搭后端（B 方案：服务端定时调 e-Stat API 抓取下发）。
**理由**：项目当前零后端，B = 从零搭后端+部署+运维，约 1.5-2 周 + 持续成本；而 e-Stat 数据月/年度更新，自动化收益极低。真要上后端应等出现别的必需后端需求（匿名案例上传/账号/推送）时顺带做。
**影响**：选 A2 静态托管替代。B 列为远期 backlog，触发条件「当出现其他后端需求时」。

### 2026-06-11 — 三参考区间卡片等高，以最高那张为准

**决定**：预测详情页「三种参考区间」三张卡片用 `IntrinsicSize.Max` 统一高度，以内容最高的那张为基准撑高其余两张。
**理由**：「正常」区间显示完整区间（如 `2026年9月中旬-11月上旬`）会换行变高，而乐观/保守是短单点，三张不等高难看。用户选择保留完整文案、靠等高对齐。
**影响**：`ScenarioCard` 的 Row 加 `IntrinsicSize.Max`，各 `Scenario` 用 `fillMaxHeight()`。

### 2026-06-11 — 采用瘦身版项目管理流程

**决定**：用 `docs/PROGRESS.md` + `docs/DECISIONS.md` + `CHANGELOG.md` 三件套，不上完整六件套（README/prd/adr/checklist/release-notes）。
**理由**：单人 + AI 结对项目，全套会烂尾成噪音。核心诉求是「跨会话记住进度」，一个永远最新的 PROGRESS 即可解决。ADR/release-notes 等以后真需要再加。
**影响**：每次有意义改动按流程：说清范围 → 更 PROGRESS/DECISIONS → 看代码 → 复用优先 → 改 → 编译 → 同步文档 → 总结。测试：UI 跳过、核心逻辑必测（见 2026-06-14 决定）。

### 2026-06-12 — 语言切换暂缓，先灰掉入口

**决定**：「我的」Tab 的「语言设置」行做成禁用态（变灰 + 不可点 + 「即将支持」标签）。底层基础设施（`LanguagePrefs` SharedPrefs + `MainActivity.attachBaseContext` 应用 locale + `findActivity().recreate()`）全部保留并已验证可用。
**理由**：Android 语言切换只对 `res/strings.xml` 里的资源字符串生效，但全 App 文字目前硬编码在 Kotlin 里。要真正生效需做「全 App 本地化」独立工程（抽字符串 + 日英翻译），MVP 阶段页面还在逐个搭，过早本地化会反复返工。用户拍板暂缓。
**影响**：`SettingsRowData.enabled` 标志控制；重新启用只需去掉 `enabled = false`。本地化工程列入上架前 backlog。

### 2026-06-12 — 我的页狗狗用专属 `DogProfile`，非欢迎页 `DogMascot`

**决定**：「我的」Tab 视觉区用新建的 `DogProfile`（正脸坐姿，对应设计 node `RnhY9`），不复用欢迎页的 `DogMascot`（站立全身）。
**理由**：两者是不同素材；最初误用 `DogMascot` 导致比例/姿态不符。
**耳朵 z-order**：绕了一圈——最终是 **head 画在耳朵之上**（body→耳朵→head→五官）。设计 JSON 里耳朵虽是 head 之后的 children，但实际渲染效果是「干净的脸、耳朵从两侧露出、内缘被头盖住」。若把耳朵画在头前，会糊到脸上、盖住眼睛，与设计不符。**以截图视觉为准，不照搬 JSON child 顺序。**
**影响**：`DogMascot.kt` 新增 `DogProfile` + `drawDogProfile`，坐标严格照搬 node `RnhY9`（186×161.2 参考框），耳朵 ±12° 旋转。

### 2026-06-11 — 首页 MOCK 用 `.copy()` 只补缺失字段

**决定**：首页预览「审查中」态时，不硬编码整份假 state，而是 `realState.copy(field = realState.field ?: mock)`，真实数据优先，仅缺失字段用假值兜底。
**理由**：让用户看到自己真实的昵称/入管/等待天数，同时能预览「审查中」视觉态。
**影响**：`HomeScreen.kt` 保留该逻辑块；`MOCK_PREVIEW=false` 时也安全。
