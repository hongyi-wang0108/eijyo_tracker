# 预测算法 & 公开数据管线方案

> 永住处理时间预测，从「固定 4-6 个月 + 拍脑袋缓冲」升级为「基于入管局真实积压/处理量的 FIFO 排队消化模型」。
> 本文是该方案的单一技术事实来源：数据源、托管方式、人工更新 SOP、算法公式、口径陷阱、降级策略。
> 决策摘要见 `DECISIONS.md`；产品规格见 `PRODUCT_PM.md` §11/§7.3.4/§14。

**最后更新**：2026-06-14（URL 定 jsDelivr；改双口径展示；待写代码）

## 0. 一句话

App 启动从 GitHub 拉一份人工维护的 JSON（各入管局每月 新受/处理量/积压 + 年度许可数），用 FIFO 排队模型算「你这件还要等多久」。无后端，人工每月跟 e-Stat 月报更新。

---

## 1. 架构：A2（托管静态 JSON + 人工更新，无后端）

```
e-Stat 月報(每月发布,约3个月延迟)
   │  ← 人工：下载 CSV → 转换脚本 → public-data.json → commit push
   ▼
GitHub 上的 public-data.json   （静态文件，非后端服务）
   │  ← App 启动拉取
   ▼
App 三层降级：
   GitHub  →（失败）→ Room 缓存（上次拉到的）→（还没有）→ APK 内置 bundled 兜底
```

- **不是后端**：没有任何服务端逻辑、不抓 e-Stat，就是一个手动维护的静态文件。
- **不用每月发版**：每月只改 JSON 文件 commit push，App 下次启动即生效。
- **永不白屏**：网络挂了用缓存，全新安装无网用 APK 内置兜底。

### 1.1 托管（已定仓库，URL 待定 1 项）

  - 与 App 仓库分离：数据每月更新不污染代码历史、不触发 CI、权限可独立。
  - 计划目录：`/public-data.json`（成品）、`/raw/`（存原始 CSV 备查）、`/scripts/`（转换脚本）。

## 2. 数据源（e-Stat 出入国管理統計）— 已下载验证

三张表，对应你下载的三个 CSV（Shift-JIS 编码）：

| # | 用途 | 表 | sid | 你的文件 | 时间粒度 |
|---|------|----|----|---------|---------|
| **主表** | 月度 受理/处理/积压（算法核心） | 表番号06 地方局別 在留資格の取得等の受理及び処理人員 | [0003449073](https://www.e-stat.go.jp/dbview?sid=0003449073) | `...230146.csv` | **月次** 2020.11~2026.3 |
| 备表 | 年度 受理/处理（交叉验证） | 地方局別 在留資格の取得等の受理及び処理人員 | [0003288730](https://www.e-stat.go.jp/dbview?sid=0003288730) | `...230058.csv` | 年次 |
| 趋势表 | 年度 永住许可数 | 国籍・地域別 地方局別 永住許可人員 | [0003289203](https://www.e-stat.go.jp/dbview?sid=0003289203) | `...230155.csv` | 年次 |

### 2.1 主表结构（关键）

- **行** = 每个 `(時間軸月次 × 在留資格 × 受理処理項目)` 组合；**列** = 各地区。
- **永住**：在留資格审查 代码 `60`（筛 `在留資格審査="永住"`）。
- **受理处理项目代码**（每个组合对应一行）：

  | 项目 | 代码 | 含义 / 算法角色 |
  |------|------|----------------|
  | 受理_総数 | 100000 | 旧受 + 新受 |
  | 受理_旧受 | 102000 | 期初积压（≈上月末未済） |
  | 受理_新受 | 103000 | 本月新申请 = λ |
  | 既済_総数 | 300000 | 本月处理完成 = μ（含许可+不许可+その他） |
  | 既済_許可 | 301000 | 本月许可 |
  | 既済_不許可 | 302000 | 本月不许可 |
  | 既済_その他 | 305000 | 本月其他（撤回等） |
  | （未済） | — | **无单列** → 用 `受理_総数 − 既済_総数` 算 |

- **地区列顺序**（主表横向）：
  `総数, 札幌管内, 仙台管内, 東京管内, 成田空港支局, 羽田空港支局, 横浜支局, 名古屋管内, 中部空港支局, 大阪管内, 関西空港支局, 神戸支局, 広島管内, 高松管内, 福岡管内, 那覇支局`
- ⚠️ 数字含千分位逗号且在引号内（如 `"55,427"`），解析要按 `","` 切分并去逗号。

### 2.2 真实样本（2026年3月·永住，验证用）

| 项目 | 全国 | 東京管内 | 横浜支局 | 大阪管内 |
|------|------|---------|---------|---------|
| 受理_総数 | 80,173 | 55,427 | 5,972 | 10,377 |
| 受理_新受(λ) | 10,950 | 5,713 | 1,290 | 1,413 |
| 既済_総数(μ) | 6,921 | 3,720 | 565 | 553 |
| 既済_許可 | 3,680 | 1,616 | 382 | 377 |
| **积压(算)** | 73,252 | 51,707 | 5,407 | 9,824 |

> 东京管内积压 51,707 / 月处理 3,720 ≈ 14 个月（含横滨；纯东京更长）。横滨处理相对快。印证了「东京特别慢」。

## 3. 口径：双口径展示（纯辖区 + 管内合计）+ 地区→enum 映射

「XX管内」= 本局 + 下属支局/出张所合计。同一个「东京」两个口径差异巨大（管内含横滨，横滨处理快会把合计拉低）。**学竞品：两个口径都存、都展示**，让用户看到差异、更可信。

- **纯辖区**（减去支局）：用户实际排的队 → **预测用这个**，也是默认展示值。
- **管内合计**（e-Stat 原始列）：仅作对比展示（「按管内整体看更快，但你所在的东京都本身要更久」）。

| `ImmigrationOffice` | 纯辖区公式（预测用） | 管内合计（对比展示） |
|---------------------|----------------------|---------------------|
| `TOKYO` | 東京管内 − 成田 − 羽田 − 横浜 | 東京管内 |
| `OSAKA` | 大阪管内 − 関西空港 − 神戸 | 大阪管内 |
| `NAGOYA` | 名古屋管内 − 中部空港 | 名古屋管内 |
| `FUKUOKA` | 福岡管内 − 那覇 | 福岡管内 |
| `YOKOHAMA` | 横浜支局 | （本身是支局，无合计） |
| `KOBE` | 神戸支局 | （同上） |
| `OTHER` | 総数（全国） | — |

- 横滨/神户本身是支局，只有单一口径。
- 机场支局（成田/羽田/関西空港/中部空港）永住量很小，但减掉更准。
- e-Stat 还有札幌/仙台/广岛/高松等局，MVP 的 enum 没有 → 归 `OTHER`（用全国総数兜底）。

## 4. JSON Schema（成品 public-data.json）

```jsonc
{
  "schemaVersion": 1,
  "dataAsOf": "2026-03",          // 主表最新月份
  "updatedAt": "2026-06-13",      // 人工更新此文件的日期
  "source": {
    "name": "出入国在留管理庁 出入国管理統計",
    "mainTableUrl": "https://www.e-stat.go.jp/dbview?sid=0003449073",
    "permitTableUrl": "https://www.e-stat.go.jp/dbview?sid=0003289203"
  },
  "standardProcessing": { "rangeLabel": "4 - 6 个月", "minMonths": 4, "maxMonths": 6 },
  "offices": [
    {
      "code": "TOKYO",
      "displayName": "东京入管",
      "regionalNote": "东京都案件量大，处理时间显著偏长。",
      "monthly": [               // 纯辖区（东京都），升序。预测 + 默认展示用
        { "month": "2026-01", "received": 4400, "processed": 3100, "pending": 46000 },
        { "month": "2026-02", "received": 4300, "processed": 3150, "pending": 45900 },
        { "month": "2026-03", "received": 4423, "processed": 3155, "pending": 45735 }
      ],
      "bureauTotal": {           // 管内合计（含横滨等支局），仅对比展示，可省略
        "label": "东京管内（含横滨等支局）",
        "monthly": [
          { "month": "2026-03", "received": 5713, "processed": 3720, "pending": 51707 }
        ]
      },
      "permitsByYear": [ { "year": 2023, "count": 12345 }, { "year": 2024, "count": 13456 } ]
    }
    // YOKOHAMA / OSAKA / KOBE / NAGOYA / FUKUOKA ...
  ]
}
```

- `received` = 新受(103000)，`processed` = 既済_総数(300000)，`pending` = 受理_総数 − 既済_総数。
- `monthly` 是**纯辖区**（按 §3 减去支局）；`bureauTotal.monthly` 是**管内合计**（e-Stat 原始列），仅对比展示，支局型 office（横滨/神户）省略此字段。
- `permitsByYear` 来自趋势表（文件3），按各局永住許可数 by year。

## 5. CSV → JSON 转换（每月重复，规划脚本）

每月人工活儿的技术核心。计划 `eijyo_tracker_data/scripts/estat_to_json.py`（实现阶段写），逻辑：

```
输入：主表 CSV(月度) + 趋势表 CSV(年度)
1. iconv SHIFT-JIS → UTF-8
2. 主表：筛 在留資格审查 == 永住(60)
3. 对每个月 × 每个项目(新受103000 / 既済総300000 / 受理総100000)，
   读出 16 个地区列的值（去千分位逗号）
4. 按 §3 映射做地区减法 → 6 个 office 的 received/processed/pending
5. 趋势表：筛永住，各局 by year 许可数 → permitsByYear（同样按 §3 拆分）
6. 组装 JSON，写 dataAsOf / updatedAt
输出：public-data.json
```

> 有脚本后，每月 SOP = 下载 3 个 CSV → 跑脚本 → 检查 diff → commit push。

## 6. 预测算法：FIFO 排队消化模型

核心：**用「你提交那刻前面排了多少人」定位队列位置，扣掉之后已处理的，剩余用最新速度估**。

### 6.1 输入
- `submitMonth`：用户提交月（`ApplicationProfile.submittedDate`，到月即可）
- `office`：用户提交地区 → 对应 `offices[code].monthly`
- `latest`：序列最后一个月（= `dataAsOf`）；`today`

### 6.2 步骤

```
1. 锚定队列位置 Q0 = monthly[submitMonth].pending   // 提交当月末积压（你前面的人）
   // 提交月早于序列起点 → 用序列最早月 pending（保守）

2. 扣减已处理 ΣP = Σ processed，从 submitMonth 之后 到 latest
   R_latest = Q0 - ΣP                               // 截至最新数据月还排你前面的

3. 外推数据延迟期（latest→today，约3个月无数据）
   gapMonths = 月差(today, latest)
   μ_latest  = monthly[latest].processed
   R_now = R_latest - μ_latest * gapMonths

4. 剩余等待
   R_now <= 0 → 「已进入可能出结果区间」
   否则      → waitMonths = R_now / μ_latest;  预计完成 ≈ today + waitMonths

5. 区间（乐观/正常/保守，绝不给精确日）
   μ_fast/μ_slow = 近 N 月 processed 的最大/最小
   乐观 = today + R_now/μ_fast;  正常 = today + R_now/μ_latest;  保守 = today + R_now/μ_slow
   // 近月 received > processed（积压在涨）→ 保守端再放宽

6. 转「上旬/中旬/下旬」标签（复用现有 DateLabels）
```

### 6.3 设计决定（可调，实现时定稿）
- **预测用「纯辖区」口径**（`monthly`）：用户在本局辖区排队，预测和默认展示用纯辖区；`bureauTotal` 管内合计仅在数据页作对比展示。
- **Q0 用提交当月末 pending**：近似你提交时前面的队列；更严谨可用旧受(期初)，差一个月量级。
- **延迟期用 μ_latest 外推**（步骤3）：让结果从「今天」起算、对用户直观。
- **永住独立队列假设**：用永住自己的积压/处理量算，假设永住是独立审查通道。实际入管资源跨资格共享 → 这是简化，与竞品一致，可接受。doc 注明，文案表述为「参考估算」。
- **「越来越准」在 A2 下打折**：靠每月人工更新；按时更新则每月重算 ΣP 增大、外推减小，确实更准；漏更新则停在快照、误差随时间长。

### 6.4 置信度（沿用扣分制，新增因子）
- 数据新鲜度 `today - dataAsOf` 越久越低；提交日期仅到月 −1；`R_now` 接近0或为负 −1；地区无数据（退回标准期间）直接「较低」。

### 6.5 边界 / 兜底
| 情况 | 处理 |
|------|------|
| office 无数据 / OTHER | 退回旧逻辑（标准 4-6 月 + 现有 buffer） |
| 提交月早于序列 | Q0 用最早月 pending（保守） |
| `R_now <= 0` | 「已进入可能出结果区间」，不报负数 |
| `μ_latest == 0` | 用近 N 月平均，避免除零 |
| 非审查中状态 | 同现状，不出预测 |

### 6.6 仍给不了的
- **个案实时进度**：入管不公开，本模型给「按平均你大概排到哪」，非真实个案 → 文案标「参考估算」。
- 非 FIFO 偏差（复杂案/补资料打断）：用区间吸收，不给精确日。
