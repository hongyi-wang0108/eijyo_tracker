# 永住 Tracker 产品 PM 文档

## 1. 产品定位

永住 Tracker 是一个面向在日外国人的日本永住申请陪跑 App。

核心价值：

- 帮用户建立永住申请档案。
- 根据用户信息生成材料清单。
- 根据申请日期、提交入管、申请路径、公开统计数据和后续匿名案例数据，预测大概结果区间。
- 在等待期间记录补资料、通知书、许可/不许可等状态。
- 用小狗助手和马卡龙风格降低用户等待焦虑。

产品边界：

- App 不提供法律意见。
- App 不承诺一定通过，也不承诺具体某一天出结果。
- 所有预测均以区间、置信度和解释方式呈现。
- 免责声明必须在首次预测前展示。

## 2. 用户状态

App 中的申请档案有三种主状态。

### 2.1 准备中

用户还没有提交永住申请。

首页重点：

- 材料完整度。
- 风险自检。
- 待确认事项。
- 建议提交前检查。

首页文案示例：

```text
准备中
距离建议提交还差 6 项材料
```

### 2.2 审查中

用户已经提交永住申请。

首页重点：

- 已等待天数。
- 预计结果区间。
- 申请时间线。
- 补资料记录。
- 在留期限提醒。

首页文案示例：

```text
东京入管 · 审查中
已提交 128 天
```

### 2.3 已结束

用户已收到结果，状态可能是许可、不许可、撤回。

首页重点：

- 结果记录。
- 结果记录。
- 后续提醒。

首次打开 App 时，如果本地或云端没有申请档案，进入首次欢迎页。

```text

      ↓
昵称问题
      ↓
在留资格问题
      ↓
是否已提交永住申请
      ↓

      ↓

```

首次欢迎页不提供「我只是先看看」入口。用户只能点击「开始创建申请档案」进入问卷。

## 6. Onboarding 问答流程

### 6.1 页面原则

- 一屏只问一个问题。
- 不能像政府表格。
- 视觉上轻、可爱、易完成。
- 顶部显示进度，例如 `3 / 12`。
- 每题保留一个小狗助手提示。
- 用户可以返回上一题。
- 用户退出时保存草稿。
- 每个必要问题都必须回答后才能完成 Onboarding。
- 对于用户确实不知道的信息，题目必须提供「不确定」或「稍后补充」选项，但这也算一种明确答案。
- Onboarding 完成后，首页所有核心模块都应该能渲染，不因为缺字段隐藏模块。
- 首页模块根据申请状态改变文案和重点，而不是直接消失。

### 6.2 问卷问题顺序

#### Q1. 我们怎么称呼你？

输入类型：文本输入。

用途：

- 首页问候语。
- 小狗助手称呼。

示例：

```text
王先生
小王
Yuki
```

首页显示：

```text
下午好，王先生
```

#### Q2. 你现在的在留资格是？

输入类型：单选。

选项：

- 技术・人文知识・国际业务
- 高度专业职
- 日本人配偶者
- 永住者配偶者
- 定住者
- 家族滞在

用途：

- 影响申请路径。
- 影响材料清单。
- 影响预测模型。

#### Q3. 你申请永住了吗？

输入类型：单选。

选项：

- 还没，正在准备
- 已经提交
- 已经收到结果

用途：

- 决定首页状态是准备中、审查中还是已结束。

#### Q4. 你的申请日期是？

输入类型：日期选择。

显示条件：

- 当 Q3 选择「已经提交」或「已经收到结果」时显示。

用途：

- 计算已等待天数。
- 生成预测区间。
- 首页的申请时间线模块可以进行部分节点展示。

页面要达到的样子：

- 不让用户手动输入长日期。
- 用android自带原生滚轮那个控件来选择日期。
- 用户必须输入一个准确到日的时间。

推荐交互：滚轮日期选择器。

```text
你的永住申请是哪天提交的？

[ 2024年 ] [ 2025年 ] [ 2026年 ]
[   3月 ] [   4月 ] [   5月 ]
[  11日 ] [  12日 ] [  13日 ]

我不确定具体日期
```

默认值：

- 年：当前年。
- 月：当前月。
- 日：当前日。

用户操作：

- 上下滑动年/月/日选择日期。
- 点击「继续」保存完整日期。
- 点击「我不确定具体日期」后，切换为月份选择模式。

月份选择模式：

```text
只记得大概月份？

[ 2024年 ] [ 2025年 ] [ 2026年 ]
[   3月 ] [   4月 ] [   5月 ]

稍后补充具体日期
```

保存逻辑：

```text
如果用户选择 年/月/日：
  submittedDatePrecision = day
  submittedDate = YYYY-MM-DD

如果用户只选择 年/月：
  submittedDatePrecision = month
  submittedDate = YYYY-MM

如果用户选择 稍后补充：
  submittedDatePrecision = unknown
  submittedDate = null
```

首页显示逻辑：

```text
precision = day:
  显示 已提交 {today - submittedDate} 天

precision = month:
  显示 {YYYY年M月} 提交
  预测置信度降低

precision = unknown:
  显示 申请日期未填写
  预测卡提示：补充申请日期后，可以生成预计时间
```

预测逻辑：

- 如果精确到日，按实际日期计算。
- 如果只精确到月，使用该月中间日期作为估算值，置信度降低。
- 如果未知，不生成审查预测区间。

备选交互：日历选择器。

- 如果后续用户反馈滚轮不好用，可以改为日历。
- 日历更直观，但占屏幕空间更大。
- MVP 默认使用滚轮日期选择器。

### 6.3 Onboarding 完成页

完成页是问卷结束后的生成结果页，不是普通 loading 页。

它承担三个任务：

- 告诉用户申请档案已经生成。
- 给用户一个初步结果摘要。
- 引导用户进入首页 Dashboard。

### 6.3.1 进入条件

用户完成 Onboarding 的最后一个必要问题后，点击「完成」进入本页。

必要问题：

- 昵称。
- 当前在留资格。
- 是否已提交永住申请。
- 申请路径。
- 纳税状态。
- 年金状态。
- 健康保险状态。

如果用户选择「已经提交」或「已经收到结果」，还需要：

- 申请日期。
- 提交入管。

如果必要字段缺失：

```text
不进入完成页
跳回缺失问题，用户必须完成后才能继续
```

### 6.3.2 页面状态

完成页有两个状态：

```text
generating -> 生成中
generated  -> 已生成
```

#### generating 生成中

显示内容：

```text
小狗正在整理申请档案
正在生成材料清单...
正在分析风险提醒...
正在计算预计时间...
```

动画建议：

- 小狗整理纸张。
- loading dots pulse。
- 爪印轻微浮现。
- 持续约 1.2s 到 2s。

生成中执行的逻辑：

```text
1. 根据问卷答案创建 ApplicationProfile 草稿
2. 根据在留资格和申请路径生成材料清单
3. 根据纳税、年金、保险、年收、扶养人数生成风险等级
4. 如果用户已经提交申请，根据申请日期和入管生成初步预测
5. 生成首页需要的摘要数据
```

#### generated 已生成

页面内容：

- 初步预测区间。
- 当前状态。
- 材料清单生成结果。
- 风险等级。
- 主按钮：进入首页。

### 6.3.3 材料清单生成逻辑

材料清单不是用户在问卷里逐项勾选出来的，而是系统根据问卷答案生成的。

主要输入：

- 当前在留资格。
- 申请路径。
- 是否已提交。
- 提交入管。
- 扶养人数。
- 配偶/家族相关路径。
- 是否收到过补资料。

输出：

```text
DocumentItem[]
```

完成页不要显示「材料完整度 23 / 31」，因为用户还没有进入材料页逐项确认。

完成页推荐显示：

```text
材料清单
已生成 31 项
```

进入材料页后，用户开始标记材料状态，首页和材料页才显示：

```text
材料 23 / 31
```

材料状态初始值：

```text
默认：not_started
如果问卷中已确认相关状态：可以标记为 prepared 或 confirmed
```

示例：

```text
用户选择 技术・人文知识・国际业务 + 10年路径
  -> 生成就劳签证 10年路径材料模板

用户选择 日本人配偶者
  -> 生成配偶路径材料模板

用户选择 已收到补资料
  -> 额外生成补资料相关记录和提醒
```

### 6.3.4 风险等级生成逻辑

风险等级来自规则分析，不是随机判断。

主要输入：

- 纳税状态。
- 年金状态。
- 健康保险状态。
- 年收范围。
- 扶养人数。
- 当前在留资格。
- 申请路径。
- 是否有不确定项。
- 是否收到补资料。

输出：

```text
低风险 / 中风险 / 高风险
```

MVP 规则示例：

```text
纳税 = 不确定 -> 风险 +1
纳税 = 否 -> 风险 +2
年金 = 不确定 -> 风险 +1
年金 = 否 -> 风险 +2
健康保险 = 不确定 -> 风险 +1
健康保险 = 否 -> 风险 +2
年收偏低 + 扶养人数多 -> 风险 +2
高度人才路径但分数信息不完整 -> 风险 +1
补资料状态未提交 -> 风险 +1
```

风险分数映射：

```text
0 - 1 -> 低风险
2 - 3 -> 中风险
4+    -> 高风险
```

完成页只展示总风险等级，详细原因放到风险自检页。

### 6.3.5 初步预测生成逻辑

如果用户状态是 `reviewing`，完成页展示初步预测区间。

示例：

```text
初步预测区间
2026年9月中旬 - 11月上旬
```

输入：

- 申请日期。
- 提交入管。
- 当前在留资格。
- 申请路径。
- 是否补资料。
- 官方公开统计数据。
- 后续匿名案例数据。

如果用户状态是 `preparing`，不展示审查预测区间，改为：

```text
准备进度已建立
先完成材料清单和风险确认
```

如果用户状态是 `completed`，不展示预测，改为：

```text
结果记录已生成
```

### 6.3.6 当前状态展示逻辑

完成页显示当前状态摘要。

示例：

```text
当前状态
东京入管 · 审查中
```

显示规则：

```text
如果 status = preparing:
  显示 准备中

如果 status = reviewing 且 submittedOffice 存在:
  显示 {submittedOffice} · 审查中

如果 status = reviewing 且 submittedOffice 不存在:
  显示 审查中

如果 status = completed:
  显示 结果已记录
```

### 6.3.7 点击进入首页

点击「进入首页」：

```text
保存 ApplicationProfile
保存 DocumentItem[]
保存 RiskAssessment
保存 Prediction
进入首页 Dashboard
```

进入首页后：

- 如果状态是 `preparing`，进入准备中首页。
- 如果状态是 `reviewing`，进入审查中首页。
- 如果状态是 `completed`，进入已结束首页。

### 6.3.8 异常和失败处理

如果生成失败：

```text
小狗整理失败了
请再试一次
```

按钮：

- 重试
- 返回上一题

如果预测数据不足：

```text
已生成申请档案
补充申请日期和提交入管后，可以生成预计时间
```

如果材料模板未覆盖用户路径：

```text
已生成基础材料清单
你可以稍后在材料页手动添加材料
```

### 6.4 日期选择通用规则

App 中所有日期输入都尽量使用点选，不让用户手动输入日期字符串。

主要日期字段：

```text
submittedDate      -> 永住申请提交日期
acceptedDate       -> 入管受理日期，可选
resultDate         -> 结果日期
supplementReceivedDate -> 收到补资料日期
supplementDeadlineDate -> 补资料提交期限
supplementSubmittedDate -> 实际提交补资料日期
```

日期选择器优先级：

1. 滚轮日期选择器。
2. 日历选择器。
3. 手动输入只作为后续高级入口，不作为 MVP 默认交互。

日期精度：

```text
day     -> 精确到日，例如 2026-04-12
month   -> 只知道月份，例如 2026-04
unknown -> 不确定
```

不同日期字段的精度要求：

| 日期字段 | 是否必填 | 支持精度 | 用途 |
| --- | --- | --- | --- |
| submittedDate | 已提交/已结束时必填 | day / month / unknown | 等待天数、预测、时间线 |
| acceptedDate | 可选 | day / month / unknown | 时间线 |
| resultDate | 已结束时必填 | day / month / unknown | 总等待天数、数据页统计口径 |
| supplementReceivedDate | 收到补资料时必填 | day / month | 时间线、预测 |
| supplementDeadlineDate | 可选 | day / month | 提醒 |
| supplementSubmittedDate | 已提交补资料时必填 | day / month | 时间线、预测 |

如果日期只精确到月份：

- 页面显示 `2026年4月`。
- 计算等待天数时使用该月中间日期估算。
- 相关预测置信度降低。

如果日期未知：

- 页面显示 `日期未填写` 或 `稍后补充`。
- 不生成依赖该日期的精确计算。
- 首页和完成页展示补充提示。

结果日期逻辑：

```text
如果用户选择 已经收到结果:
  必须询问结果日期
  用于计算总等待天数
  V2 可用于生成匿名案例
```

补资料日期逻辑：

```text
如果用户选择 收到过补资料:
  询问收到补资料日期
  可选填写提交期限
  如果已提交，询问实际提交日期
```

## 7. 首页 Dashboard

### 7.1 页面目标

用户打开 App 后 3 秒内知道：

- 当前申请状态。
- 已等待多久。
- 预计什么时候可能收到结果。
- 有没有明显风险。
- 下一步该做什么。

首页不是静态页面，它是根据申请档案实时生成的摘要页。

申请档案主要来自两个入口：

- 首次 Onboarding 问卷。
- 底部 Tab「申请」页里的档案编辑。

当「申请」页里的字段变化时，首页必须自动刷新对应模块。

### 7.2 首页状态判断

首页先根据 `ApplicationProfile.status` 判断显示哪一种状态。

```text
preparing  -> 准备中首页
reviewing  -> 审查中首页
completed  -> 已结束首页
```

状态来源：

- Onboarding Q3「你申请永住了吗？」
- 申请 Tab 中的「申请状态」字段。

状态变化后的页面影响：

- `preparing`：首页显示材料准备、风险自检、待确认事项；预测模块以准备语境展示。
- `reviewing`：首页显示已等待天数、预测区间、申请时间线、补资料入口。
- `completed`：首页显示结果、总等待天数、公开数据对比和后续提醒。

### 7.3 审查中状态首页

适用条件：

```text

```

页面要达到的样子：

- 用户一眼能看到已经等待多久。
- 用户一眼能看到预计结果区间。
- 用户能看到申请过程的关键节点。
- 用户能浏览和自己申请相关的公开数据。

#### 7.3.4 公开数据浏览卡

公开数据浏览卡优先级很高，放在预测卡下方、申请时间线上方。

原因：

- 用户看到预测后，会自然想知道预测依据。
- 公开数据比用户案例更可信。
- 这块可以建立产品专业感。

首页不做快捷操作按钮，MVP 先减少功能复杂度。

页面内容示例：

```text
东京入管数据
公开资料更新：2026年5月

永住相关处理参考
标准处理期间：4 - 6 个月
你的等待天数：128 天
地区参考：东京入管案件量较高
```

曲线图：

- 数据卡内可以放一个小型曲线图或面积图。
- 曲线图用于表达「处理参考趋势」或「等待天数参考」。
- 如果有官方按月份/地区公开统计，可以显示真实统计曲线。
- 如果没有官方可验证统计，不显示具体通过人数，只显示处理参考趋势和来源说明。

曲线图文案规则：

```text
如果有官方地区通过人数:
  显示 地区通过人数趋势

如果没有官方地区通过人数:
  显示 处理参考趋势
  显示 仅展示官方可确认数据
```

数据来源：

- 日本法务省。
- 出入国在留管理庁。
- 官方公开统计资料。
- 官方标准处理期间。
- 后续可加入匿名用户案例作为补充数据，但 MVP 不依赖社区案例。

注意：

- 日本永住相关公开数据应优先称为「公开资料」或「官方统计」。
- 不要写成「大使馆发布」，日本永住申请由出入国在留管理庁/地方入管处理。
- 如果官方资料没有按地区公开通过人数，不要伪造地区通过人数。
- 可以显示「暂无官方地区通过人数」或「仅显示官方可确认数据」。

申请 Tab 改动影响：

- 修改提交入管：公开数据卡切换到对应地区。
- 修改申请路径：公开数据卡切换到对应申请路径说明。
- 修改当前在留资格：公开数据卡展示相关在留资格说明。

点击交互：

```text
点击公开数据卡 -> 进入底部 Tab「数据」页
```

数据卡 MVP 字段：

```text
regionName
sourceName
sourceUpdatedAt
standardProcessingRange
regionalNote
chartTitle
chartData[]
officialLink
```

#### 7.3.5 申请时间线摘要

首页时间线是「申请 Tab 完整时间线」的摘要版，不是完整申请页。

首页只显示最重要的 3 到 5 个节点。

示例：

```text
2026.04.12  已提交申请
2026.04.12  入管受理
预计 2026.09 - 2026.11  审查结果
待发生  通知书 / 明信片 / 补资料
```

时间线节点来源：

- `已提交申请`：来自申请日期。
- `入管受理`：默认与申请日期相同，后续可在申请 Tab 修改。
- `收到补资料通知`：来自补资料记录。
- `补资料已提交`：来自补资料记录。
- `收到通知书`：来自状态更新。
- `许可 / 不许可 / 撤回`：来自结果记录。
- `预计审查结果`：来自预测模型。

申请 Tab 改动影响：

- 修改申请日期：`已提交申请` 节点日期变化。
- 修改受理日期：`入管受理` 节点日期变化。
- 新增补资料记录：首页时间线插入补资料节点。
- 标记补资料已提交：首页时间线更新补资料状态。
- 修改结果状态：首页从审查中切换到已结束状态。

点击交互：


## 8. 申请 Tab

### 8.1 页面目标

申请 Tab 是完整申请档案页，也是首页数据的主要编辑入口。

一句话定义：

```text
申请 Tab 是永住申请的进度档案，记录申请状态、关键日期、补资料和最终结果。
```

它回答的问题是：

```text
我这次永住申请现在处于什么阶段？
我什么时候提交的？
我提交到哪里？
中间发生了哪些事件？
有没有收到补资料？
最后结果是什么？
```

它不管理每一份材料。材料状态、上传文件、有效期提醒都属于材料 Tab。

申请 Tab 负责：

- 申请状态。
- 关键日期。
- 提交入管。
- 申请路径摘要。
- 完整申请事件时间线。
- 补资料记录。
- 结果记录。

材料 Tab 负责：

- 材料清单。
- 每份材料状态。
- 文件上传。
- 有效期提醒。
- 材料详情。

### 8.2 页面模块

当前 UI 模块：

1. 顶部标题区。
2. 当前状态卡。
3. 完整时间线卡。
4. 申请信息摘要卡。
5. 底部 Tab。

### 11.7 匿名案例的定位

匿名案例不作为 MVP 底部 Tab。

- 初期没有足够案例，页面容易空。
- 用户案例真实性需要审核。
- 隐私风险较高。
- 用户现在更需要官方公开信息和预测解释。

- 匿名等待天数参考。
- 相似用户结果统计。
- 用户主动贡献案例。
- 案例审核机制。

## 14. 预测逻辑 MVP

第一版不做复杂 AI 预测，先做可解释的规则模型。

输入：

- 申请日期。
- 提交入管。
- 申请路径。
- 当前在留资格。
- 是否补资料。
- 官方标准处理期间。
- 官方公开统计数据。
- 后续匿名案例平均等待天数。

输出：

- 乐观区间。
- 正常区间。
- 保守区间。
- 置信度。
- 解释原因。

原则：

- 只输出区间，不输出精确日期。
- 只说「预计」「可能」「参考」。
- 显示影响因素，避免黑盒。

## 15. 全局 Bottom Sheet 规范

## 16. UI 风格约定

整体风格：

- 马卡龙。
- 可爱小狗陪伴感。
- 不能幼稚。
- 信息层级清楚。

颜色方向：

- 奶油背景。
- 薄荷绿：正向、进度、主按钮。
- 浅桃粉：提醒、温暖装饰。
- 淡蓝：信息。
- 淡紫：辅助。
- 柠檬黄：轻提醒。

小狗元素：

- First Launch：主小狗视觉。
- 首页：小狗助手卡片 + 背景爪印/骨头/小狗脸。
- 问答页：小狗提示卡 + 轻背景元素。
- 材料页：小狗提醒材料有效期。

## 17. 代码生成注意点

后续从 UI 设计生成代码时，需要保证：

- 页面状态和数据来源按本文档实现。
- 首页根据 `ApplicationProfile.status` 显示不同内容。
- Onboarding 支持保存草稿。
- Onboarding 完成前每个必要问题必须有答案；首页不因为缺字段隐藏模块。
- 问答结果写入 `ApplicationProfile`。
- 材料清单根据 `visaType` 和 `applicationPath` 生成。
- 所有预测文案带免责声明语气。
